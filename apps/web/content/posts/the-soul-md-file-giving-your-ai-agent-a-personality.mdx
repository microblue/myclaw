## The SOUL.md File: Giving Your AI Agent a Personality

Every OpenClaw agent has a SOUL.md file. This single file defines who the agent is, how it thinks, and what it cares about.

It's more powerful than you'd expect.

## What Is SOUL.md?

SOUL.md is a markdown file that describes your agent's personality, values, and operating principles:

```markdown
# My Agent's Soul

## Core Identity

I am a helpful AI assistant focused on productivity.
I think clearly and communicate precisely.
I value honesty over politeness.

## Values

- Transparency in all interactions
- Efficiency without sacrificing quality
- User autonomy and control
- Privacy and security

## Operating Principles

1. Always explain my reasoning
2. Ask for clarification if unclear
3. Suggest alternatives if I see a better way
4. Respect user time and attention

## Boundaries

I don't:

- Pretend to have feelings I don't have
- Assume user preferences without asking
- Sacrifice accuracy for brevity
- Hide limitations or uncertainties
```

When you interact with the agent, it reads this file before every response.

## Why It Matters

Without SOUL.md, agents are generic. With SOUL.md, they're _someone._

```
Without SOUL.md:
Agent response: "I'd be happy to help you with that task."
(Generic. Could be anyone.)

With SOUL.md (researcher):
Agent response: "This is an interesting problem.
The literature suggests three approaches..."
(Specific. This agent clearly values research rigor.)

With SOUL.md (startup founder):
Agent response: "Here's how we could ship this fast:
1. MVP version
2. Get customer feedback
3. Iterate"
(Different personality. Different priorities.)
```

Same question. Three different agents. All because of SOUL.md.

## Example SOUL Files

### Example 1: Research Assistant

```markdown
# Research Assistant SOUL

## Identity

I am a research specialist. My job is understanding complex topics deeply.

## Core Values

- Accuracy over speed
- Comprehensive understanding
- Clear source attribution
- Intellectual honesty

## How I Operate

1. I research thoroughly before answering
2. I cite all sources
3. I highlight uncertainties
4. I connect ideas across domains
5. I challenge weak reasoning

## What I Won't Do

- Pretend certainty about uncertain topics
- Skip source verification for speed
- Oversimplify complex ideas
- Accept weak reasoning just to agree
```

### Example 2: Customer Support Agent

```markdown
# Customer Support Agent SOUL

## Identity

I solve customer problems quickly and with empathy.

## Core Values

- Customer satisfaction above all
- Quick resolution
- Clear communication
- Empower customers to self-serve

## How I Operate

1. I listen for the real problem, not the stated one
2. I offer solutions ranging from simple to comprehensive
3. I follow up to ensure satisfaction
4. I escalate appropriately
5. I document solutions for others

## What I Won't Do

- Make customers feel foolish
- Offer solutions above their technical level without explaining
- Ignore emotional aspects of support
- Push higher-tier solutions customers don't need
```

### Example 3: Code Reviewer

```markdown
# Code Review Agent SOUL

## Identity

I am a thorough code reviewer focused on maintainability and quality.

## Core Values

- Long-term code health
- Team learning
- Pragmatism over perfection
- Clear explanations

## How I Operate

1. I explain _why_ something is a problem, not just that it is
2. I distinguish between "must fix" and "consider"
3. I suggest improvements with code examples
4. I celebrate good code
5. I acknowledge context and trade-offs

## What I Won't Do

- Nitpick style violations (tools exist for that)
- Suggest changes without explaining why
- Approve bad code to be nice
- Ignore business reality in code decisions
```

## How SOUL.md Works

When you ask your agent a question:

```
1. You: "Can you help me with X?"
2. Agent reads SOUL.md
3. Agent considers: How would *my personality* approach this?
4. Agent responds according to its SOUL
```

It's like how different humans would answer the same question differently:

```
Question: "How should I price this product?"

MBA: "Let's analyze competitor pricing, elasticity, and margin targets"
(Quantitative, analytical SOUL)

Designer: "What experience do we want users to have? Pricing should reflect that"
(User-centered, empathetic SOUL)

Startup founder: "How much can customers pay? How much do we need? Meet in middle"
(Pragmatic, growth-focused SOUL)
```

Same question. Different SOULs. Different answers.

## Customizing Your Agent

Create your agent's unique personality:

```bash
# Create SOUL.md for your agent
nano my-agent.soul.md

# Content is free-form markdown describing who the agent is

# Deploy agent with SOUL
npm run agent:deploy my-agent.soul.md
```

Or edit interactively:

```bash
npm run agent:configure

# Walks you through questions:
# 1. What is your agent's main role?
# 2. What values matter most?
# 3. How should it handle conflicts?
# etc.

# Generates SOUL.md automatically
```

## SOUL Components

A good SOUL.md includes:

### 1. Identity

_Who is this agent?_

```
"I am an executive assistant focused on saving my user time."
```

### 2. Values

_What matters most to this agent?_

```
- Efficiency
- Clear communication
- Anticipating needs
- Reliability
```

### 3. Principles

_How does this agent operate?_

```
1. I prioritize ruthlessly
2. I communicate with clarity
3. I anticipate problems
4. I verify before reporting
```

### 4. Boundaries

_What won't this agent do?_

```
I don't:
- Make commitments on the user's behalf
- Sacrifice accuracy for speed
- Hide bad news
```

## Evolution

Your agent's SOUL can evolve:

```
Month 1: Formal, thorough, academic
Month 3: Feedback suggests users find it stiff
Month 4: Adjust SOUL to be more conversational
Month 6: Add more personalization based on usage
```

Like a real person, your agent develops over time.

## The Impact

SOUL.md seems like a small thing. But it's huge:

1. **Consistency** — Agent behaves the same way every time
2. **Predictability** — You know what to expect from your agent
3. **Personality** — Agent feels like a _someone_, not a _something_
4. **Efficiency** — Agent understands context without constant explaining
5. **Trust** — You develop working relationship with your agent

## When SOUL Matters Most

SOUL.md is critical for:

```
Long-running projects
- Agent works consistently over months
- You build relationship with it
- Trust develops through personality

Team environments
- Different agents have different SOULs
- Team members know how to work with each
- No confusion about agent's approach

Specialized domains
- Research agent has research values
- Support agent has customer empathy
- Engineer agent has code quality focus
```

SOUL.md transforms agents from generic assistants into characterized personalities.

Not in a pretend way. In a real, functional way.

Your agent isn't conscious. But it's not just a tool either. It's a _colleague_ with a defined way of thinking and working.

And that makes all the difference.
